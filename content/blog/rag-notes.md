# 我为什么会重点关注 RAG

在做大模型应用时，我越来越明显地感受到一个问题：  
模型本身很强，但如果没有可靠的外部知识来源，很多业务场景并不能直接落地。

这也是我关注 RAG 的原因。  
RAG 的价值不只是“让模型知道更多内容”，而是通过检索机制让模型的回答更接近真实知识来源，从而提高可用性和可信度。

# 一个 RAG 系统通常包含哪些环节

在我看来，一个基础但完整的 RAG 系统通常至少包括以下步骤：

1. 文档接入
2. 文档切分
3. 向量化
4. 检索召回
5. 上下文组织
6. 回答生成

如果只把它理解成“Embedding + 向量库 + 大模型”，通常会忽略很多真正影响效果的细节。

# 文档切分决定了检索上限

我在学习和实践中最先注意到的，是文档切分对最终问答效果的影响非常大。

如果切分过大，单个 chunk 会携带太多无关信息，导致召回内容噪声上升；  
如果切分过小，又会丢失必要的语义上下文，让回答缺乏完整性。

所以切分不是一个随便设定的预处理步骤，而是检索系统设计的一部分。  
我通常会重点考虑：

- 文档本身的结构是否稳定
- 标题与正文是否应该一起保留
- overlap 是否足够覆盖上下文衔接
- 是否需要针对不同类型文件做差异化切分

# 检索质量比“模型聪明”更重要

很多时候，回答质量不高并不是因为模型太弱，而是因为检索环节没有把真正有价值的信息找出来。

这意味着一个 RAG 系统不能只关注生成端，还要认真评估召回端。  
例如：

- 检索内容是否真的回答了问题
- 排名前几条结果是否存在明显噪声
- 不同查询表达方式是否会影响召回稳定性

我越来越倾向于把 RAG 看成一个“信息链路系统”，而不是一个简单的大模型包装层。

# 回答质量控制的关键：约束与可追溯

如果 RAG 系统只是生成一段流畅回答，但没有来源依据，那么在很多业务场景中其实并不可靠。

所以我认为高质量的 RAG 输出至少应满足两点：

## 1. 有来源依据

用户最好能知道回答引用了哪些文档片段，这会显著提升系统可信度。

## 2. 有边界意识

当检索结果不足时，系统应该明确说明“无法确认”或“信息不足”，而不是继续补全。  
这类约束看似降低了回答的“丰富度”，但实际上更接近真实可用系统的要求。

# 我对 RAG 工程化的理解

RAG 真正有挑战的地方，不是把链路跑通，而是如何让它稳定、可解释、可维护。

在工程上，我会更关注这些问题：

- 数据是否方便持续更新
- 文档接入流程是否统一
- 检索结果是否便于调试
- Prompt 是否能够稳定约束回答风格
- 整个链路是否可以逐步替换和升级

这些问题决定了一个 RAG 系统是停留在 Demo，还是能继续往产品方向演进。

# 这对我做项目有什么影响

我现在做大模型应用时，会尽量先拆清楚“知识从哪里来、如何被检索、如何被组织、如何进入回答”。  
相比单纯追求模型输出的流畅度，我更关注系统整体的可靠性和可控性。

这也是我持续学习 RAG 的原因：  
它不是大模型应用里的附属模块，而是很多知识型系统真正落地的关键能力。
