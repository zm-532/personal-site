# 我为什么会做这个项目

在学习大模型应用的过程中，我越来越明显地感受到一个问题：

很多项目都在强调“问答能力”“智能助手”“自动生成”，但如果离开具体场景，这些能力很容易停留在演示层面，难以真正体现业务价值。

我想做一个更接近实际训练场景的项目，于是就有了 `anti-fraud-agent`。

这个项目的目标并不是做一个泛泛的聊天机器人，而是构建一个**面向大学生用户的反诈模拟训练平台**。  
它聚焦一个非常具体、也非常常见的诈骗场景：

**虚假兼职刷单诈骗。**

相比普通聊天，这类场景更适合做成结构化训练系统，因为它天然具备：

1. 明确的流程阶段
2. 可观察的用户决策行为
3. 可量化的风险变化
4. 可输出的训练结果与反馈

这让我意识到，一个好的 LLM 项目，不只是“能对话”，而是应该具备：

- 场景约束
- 状态控制
- 行为记录
- 结果评估

这也是我设计这个项目时最关注的几个点。

---

# 这个项目要解决什么问题

`anti-fraud-agent` 是一个基于 **LLM Agent + FSM（有限状态机）** 的模拟训练项目，核心目标是让用户在一个相对安全的环境里，体验诈骗话术是如何一步步推进的，并在过程中识别风险。

项目主要完成以下几件事：

1. 使用大模型扮演诈骗角色
2. 使用 FSM 控制诈骗流程的阶段推进
3. 记录用户在不同阶段的关键行为
4. 对用户行为进行风险评分
5. 在训练结束后输出结构化报告

也就是说，这个项目不只是“生成回复”，而是要形成一个完整的训练闭环：

**诈骗模拟 -> 用户应对 -> 风险识别 -> 行为评估 -> 训练报告**

# 为什么选择 LLM Agent + FSM

我在设计这个项目时，没有把整个对话完全交给大模型自由发挥，而是选择了：

**LLM Agent 负责生成话术，FSM 负责控制流程。**

这样做的原因很明确。

## 1. 只用大模型，自由度太高

如果把诈骗对话完全交给 LLM，它当然能生成很多“像样的话术”，但问题也很明显：

- 对话推进可能不稳定
- 场景阶段不清晰
- 很难做行为评分映射
- 报告输出缺少结构支撑

换句话说，模型会“说话”，但不一定“按训练目标说话”。

## 2. FSM 天然适合诈骗流程建模

诈骗训练场景其实很适合用有限状态机来抽象，比如：

- 初始接触
- 抛出诱饵
- 建立信任
- 引导操作
- 要求转账
- 制造压力
- 结束总结

这些阶段本身就是一个有序流程。  
FSM 的价值就在于：

- 让对话阶段更清晰
- 让每一步更容易被记录
- 让训练路径更容易扩展
- 让后续评估规则更容易落地

所以我最后采用的是一种比较实用的组合方式：

**FSM 决定“现在处于哪个阶段”，LLM 决定“这个阶段怎么说”。**

# 项目整体设计思路

我把项目拆成了几个相对独立的模块，每个模块只负责一类事情，这样后续迭代时会更清晰。

## 1. 场景配置层

这一层主要定义诈骗训练场景的基本信息，比如：

- 场景 ID
- 场景标题
- 开场白
- 风险信号
- 训练说明

对应模块包括：

- `scenario_config.py`
- `config.py`

这一层的作用是把“训练内容”从代码逻辑里分离出来，方便以后扩展多个诈骗场景，比如：

- 冒充公检法
- 虚假贷款
- 虚假中奖
- 冒充客服退款

---

## 2. FSM 状态控制层

这一层负责描述诈骗过程的“阶段”以及阶段之间如何流转。

对应模块包括：

- `states.py`
- `events.py`
- `transitions.py`

例如，当前状态可能是：

- `BAIT`
- `TRUST_BUILD`
- `TASK_GUIDANCE`
- `PAYMENT_REQUEST`
- `PRESSURE`

而触发状态变化的事件可能是：

- 用户继续回应
- 用户接受任务
- 用户拒绝
- 用户追问证明
- 会话结束

通过这层，整个模拟流程不会完全失控，而是始终围绕训练目标推进。

---

## 3. 对话控制层

这一层是项目的核心协调器。

对应模块包括：

- `dialogue_controller.py`
- `session.py`

它主要负责：

- 创建训练会话
- 记录每轮对话
- 接收用户输入
- 调用角色 Agent 生成回复
- 更新会话状态
- 为后续评估提供上下文

我把每次训练抽象成一个 `TrainingSession`，这样整个系统可以围绕“会话”来组织数据，而不是零散地处理文本。

---

## 4. 角色生成层

这一层负责让大模型“扮演诈骗角色”。

对应模块包括：

- `prompt_builder.py`
- `role_agent.py`

这里的思路不是让模型随便聊天，而是结合：

- 当前诈骗场景
- 当前 FSM 状态
- 用户上一轮输入

来动态构造提示词，再生成下一步的话术。

这种设计有两个明显好处：

1. 回复更贴近当前阶段
2. 后续更容易做风格控制与安全限制

---

## 5. 行为跟踪与风险评估层

这一层决定项目是否真正具备“训练价值”。

对应模块包括：

- `behavior_tracker.py`
- `risk_rules.py`
- `evaluator.py`

系统会记录用户在关键节点的行为，比如：

- 是否相信高收益承诺
- 是否愿意继续操作
- 是否愿意转账
- 是否愿意提供敏感信息
- 是否识别并中止风险对话

这些行为会映射到不同的风险分值，最后汇总出训练结果。

这部分非常关键，因为如果没有行为跟踪，那么整个项目就只剩下“聊一段诈骗对话”，很难体现训练系统的价值。

---

## 6. 报告生成层

训练结束之后，系统需要把过程转成一个用户能理解、也能复盘的结果。

对应模块是：

- `report_generator.py`

这个报告未来会重点包含：

- 会话摘要
- 风险评分
- 用户关键决策记录
- 高风险行为提示
- 改进建议

这一步是我认为很适合博客和面试展示的部分，因为它能明显体现：

**项目不是只会“生成”，而是能“评估并反馈”。**
