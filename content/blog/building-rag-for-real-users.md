# 为什么 RAG 不是“接上向量库”就结束

很多 RAG 项目的第一版都能跑起来，但真正进入可用阶段后，问题往往不在模型本身，而在检索和上下文组织。

我在做文档问答系统时，最先关注的不是模型输出有多“聪明”，而是：

- 检索到的内容是否真的相关
- 回答是否能回到原文依据
- 不同类型文档是否能被统一处理

# 我重点做的三件事

## 1. 先优化文档切分

如果切分粒度过粗，召回内容会包含太多噪声；如果切得太细，语义上下文又容易断裂。  
所以 `chunk size` 和 `chunk overlap` 不是简单参数，而是直接影响问答效果的核心设计。

## 2. 把“可追溯性”放进产品体验

用户并不只需要一个答案，更需要知道答案来自哪里。  
因此在系统里返回引用来源，比单纯提高一点生成流畅度更重要。

## 3. 用约束减少幻觉

我更倾向于把提示词当作“行为边界”而不是“文案模板”。  
明确要求模型仅基于检索内容作答，在信息不足时直接说明，而不是自由补全。

# 对我自己的启发

RAG 的价值不只是调用大模型，而是在业务语境下构建一条可控的信息链路。  
这也是我做大模型应用时最关注的方向：让系统不仅能回答，还要回答得可靠、可解释、可维护。
